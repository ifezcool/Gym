Problem 1 — Duplicate logout-btn ID in ps_contact_layout.
In the current ps_contact_layout definition, there are two dbc.Button components with id="logout-btn". One is in the header dbc.Col on the right side of the top row, and one is inside the _nav_card(...) body. suppress_callback_exceptions=True is set in app.py, so no error is thrown — but having duplicate component IDs causes Dash's callback graph to behave unpredictably and can silently prevent other callbacks on the same layout from firing, including search_enrollee. Remove the dbc.Button("Logout", id="logout-btn", ...) from inside the _nav_card(...) body in ps_contact_layout only. Keep the one in the header row. Do the same audit for ps_provider_layout and ps_claims_layout — check if they also have duplicate logout-btn IDs introduced in the same update and remove the duplicates from the nav card bodies there too.
Problem 2 — search_enrollee callback fires on auth-store changes and wipes the search result.
The search_enrollee callback has Input("auth-store", "data") as one of its inputs. This means the callback fires whenever the auth store updates — including when data finishes loading after login — with n_clicks=None and enrollee_id=None, which causes the callback to return only the overview cards and clear any existing search result the user was looking at. Change Input("auth-store", "data") to State("auth-store", "data") in the search_enrollee callback signature. The auth data is only needed for the authentication guard at the top of the function body — it does not need to trigger the callback. The existing inputs that should remain as Input are contact-search-button.n_clicks and data-ready-store-ps.data only.
Constraints:

Do not change any logic inside search_enrollee — only change the Input/State classification of auth-store in its decorator
Do not change the logout callback or the logout-btn ID used in the header rows of any layout
Do not touch ps_services_layout — it does not have a nav card with a logout button
suppress_callback_exceptions=True is set — removing the duplicate IDs will not cause any errors but will immediately restore correct callback routing for the affected layouts
After fixing Problem 1, verify there is no other component ID duplicated across the static layout variables (ps_login_layout, ps_provider_layout, ps_claims_layout, ps_contact_layout) since all four can be active in the DOM simultaneously depending on auth state — any shared ID across them will cause the same class of silent failure
