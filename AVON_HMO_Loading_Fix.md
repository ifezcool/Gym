Issue 1: Table Layout — Second Column Has No Borders/Structure
In the search_enrollee callback in pages/combined.py, the booking details table is built by transposing a DataFrame (booking = member_df.loc[...].transpose()). Each row then becomes a (label, values) pair where values[0] is the cell content. The table rows are built with proper styling on the label <td> (left column), but the value <td> (right column) only has color and padding — no backgroundColor, no border, no borderRight. The left column gets alternating #FFFFFF/#F5F3FF backgrounds and a borderRight, but the right column has none of that, making it appear loose and unbordered.
Fix: In the table_rows loop inside search_enrollee, update the second html.Td (the value cell) to include the same bg_color background as the label cell, add borderBottom: 1px solid #F3F4F6, and add border: 1px solid #F3F4F6 so it visually matches the left cell and the row looks unified. Both cells should share the same background per row.

Issue 2: Policy Year Dropdown — Range May Miss Previous Year
In query_ps_q2, the WHERE clause is:
WHERE PolicyEndDate >= DATEADD(MONTH, -3, GETDATE())
This only fetches records where PolicyEndDate is within the last 3 months. If a member's previous policy year ended more than 3 months ago (e.g., a policy that ended in June 2024 viewed in April 2026), that record will not be in store-q2 at all, so the dropdown options built from df['policy_year_str'] will simply never include it.
The update_form_on_policy_year callback correctly reads from store-q2 and filters by policy_year_str, so the logic itself is fine — the data just isn't there to begin with.
Fix: Change query_ps_q2 to remove the date restriction entirely, or extend it to at least 24 months back (DATEADD(MONTH, -24, GETDATE())). Since this is a provider portal used by internal staff, returning more records is acceptable. The recommended change is:
WHERE PolicyEndDate >= DATEADD(MONTH, -24, GETDATE())
This ensures records from the prior policy year (e.g., Apr 2024–Mar 2025) are included when staff are viewing in Apr 2026. If you want all historical records with no cutoff at all, simply remove the WHERE PolicyEndDate clause entirely — but -24 months is a safe balance between completeness and query performance.
